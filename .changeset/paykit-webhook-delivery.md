Add outgoing Ed25519 webhook delivery. Set `webhookUrl` and orvacon POSTs a signed lifecycle event on every persisted transition (authorize / capture / refund / inbound webhook). The delivered `WebhookEvent` is minimal and stable — `{ id: evt_…, type, createdAt, data: { paymentId, status, amount, gatewayReference, occurredAt } }` — and deliberately omits the gateway's raw payload, so neither PII nor unbounded size crosses the wire. Each delivery is signed over `id.timestamp.payload` (cryptokit `signWebhook`) and carries `orvacon-id` / `orvacon-timestamp` / `orvacon-signature` headers that map 1:1 to `verifyWebhook` on the receiver. Delivery is fire-and-forget so the payment flow never blocks: transient failures (5xx / 429 / 408 / network) retry with exponential backoff + full jitter (the `retry` config is now applied), any other 4xx is permanent, and an exhausted delivery surfaces through `onError`. New `drainWebhooks()` awaits in-flight deliveries for short-lived runtimes (scripts, serverless handlers). `webhookSigningKey` is now parsed with cryptokit's `parseSecretKey` at construction, so a malformed key fails fast at setup rather than at the first delivery. Adds the `EventId` brand (`evt_…`) with `eventId` / `generateEventId`, and exports `WebhookEvent`, `RetryConfig`, and the `WEBHOOK_*_HEADER` constants. There is no persistent outbox in v1 — an in-flight retry is lost on a crash or serverless freeze; durable redelivery lands with the event-storage adapter methods.
